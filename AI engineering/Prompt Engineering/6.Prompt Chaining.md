## What is prompt chaining?

Instead of asking the model to do everything in one big prompt, you **break the task into smaller focused steps**. The output of each step becomes the input of the next one — like an assembly line.

Each station in the line does one job really well and passes its result forward.

---

## Why not just use one big prompt?

You can try, but here's what happens:
- The model has to juggle too many instructions at once — quality drops on each part
- If one part of the output is wrong, the whole thing is wrong
- Hard to debug — you don't know *which step* went wrong
- Harder to improve — you'd have to rewrite everything

With chaining, each step is small, focused, and testable on its own.

## Example 1 — blog post writer chain

Three steps: generate outline → write the draft → polish the tone.

```python
from groq import Groq
client = Groq(api_key="your-groq-api-key")

def call_llm(system: str, user: str) -> str:
    """Helper function — keeps chain code clean and readable"""
    response = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        max_tokens=800,
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": user}
        ]
    )
    return response.choices[0].message.content


# ── Step 1: Generate an outline ──────────────────────────────
topic = "Why beginners should learn Python before any other language"

outline = call_llm(
    system="You are a content strategist. Create clear, structured blog outlines.",
    user=f"Create a 4-section outline for a blog post about: {topic}"
)

# ── Step 2: Write the full post using the outline ────────────
# Output of step 1 becomes input of step 2
draft = call_llm(
    system="You are a skilled blog writer. Write engaging, beginner-friendly content.",
    user=f"""Write a full blog post using this outline:

{outline}

Original topic: {topic}
Keep it conversational and around 400 words."""
)

# ── Step 3: Polish the tone ──────────────────────────────────
# Output of step 2 becomes input of step 3
final = call_llm(
    system="""You are a professional editor.
Improve clarity and flow. Make it sound natural and confident.
Do not change the meaning or add new information.""",
    user=f"Edit and improve this blog post:\n\n{draft}"
)

print(final)
```

## Example 2 — customer review processing chain

Takes a raw review and runs it through 3 focused steps: extract facts → classify urgency → draft a reply.

```python
from groq import Groq
client = Groq(api_key="your-groq-api-key")

def call_llm(system: str, user: str) -> str:
    response = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        max_tokens=400,
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": user}
        ]
    )
    return response.choices[0].message.content


raw_review = """
Ordered the blue jacket last Tuesday. It arrived damaged —
the zipper is broken. Really disappointed, I needed it for a trip.
Tried calling support but nobody picked up.
"""

# ── Step 1: Extract key facts ────────────────────────────────
facts = call_llm(
    system="Extract key facts from customer reviews. Be brief and structured.",
    user=f"""Extract the key facts from this review as bullet points:
- Product
- Problem
- Customer emotion
- Action already taken

Review: {raw_review}"""
)

# ── Step 2: Classify urgency ─────────────────────────────────
urgency = call_llm(
    system="You classify customer support tickets. Reply with only: LOW, MEDIUM, or HIGH",
    user=f"Based on these facts, classify the urgency:\n\n{facts}"
)

# ── Step 3: Draft a support reply ────────────────────────────
reply = call_llm(
    system="""You are a customer support agent.
Write empathetic, helpful replies. Acknowledge the issue first.
Offer a clear next step. Keep it under 100 words.""",
    user=f"""Write a support reply using this context:

Facts: {facts}
Urgency level: {urgency}
Original review: {raw_review}"""
)

print(reply)
```

## When to use chaining

| Situation | Use chaining? |
|---|---|
| Task has multiple distinct steps | Yes |
| Output of one step decides the next | Yes |
| You need to inspect intermediate results | Yes |
| Simple single-step task | No — one prompt is fine |
| Steps are too connected to separate | No — keep it one prompt |

---

## Key Insight

> Each step in a chain should do **one thing only**.

If you find yourself writing "and also..." in a step's instructions, that's a sign it should be split into two steps. The more focused each step, the better the overall result.

A good chain is like good code — small functions that do one thing well, composed together to solve a bigger problem.
